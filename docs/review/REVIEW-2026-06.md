# Revisión integral de OpenTermX — 2026-06

> Sesión de **solo lectura** (no se modificó código; el único archivo creado es este
> reporte). Cinco ejes: correctitud, legibilidad, arquitectura, seguridad, performance.
> Severidades: **Critical / Requerido / Nit / Optional / FYI**.
> Invariante central auditada: *ningún comando que modifique configuración de un equipo
> se ejecuta sin pasar por el ApprovalGate (aprobación humana)*.

---

## 1. Resumen ejecutivo

**Veredicto: APPROVE con cambios requeridos.**

La arquitectura es sólida y la **invariante central se sostiene**: todo camino que inyecta
comandos a un equipo pasa por aprobación humana (`propose_commands`, `run_macro`,
`open_session`) o por la whitelist read-only estricta (`run_readonly_command`,
`refresh_device_fingerprint`, telemetría interna). **No se encontró ninguna ruta de
ejecución que eluda el gate → no hay hallazgos Critical.**

Lo que sí requiere atención antes de considerarlo "aprobado por un staff engineer" es la
**integridad de la auditoría** (CSV sin concurrencia segura ni defensa contra inyección de
fórmulas, con datos de equipos no confiables), una **fuga de la malla de roles**
(auto-declarada por header), un **falso "SAFE" del semáforo** para verbos no catalogados, y
**higiene de repo** (scratch files commiteados, sin LICENSE, sin CI). Ninguno rompe la
invariante central; sí degradan trazabilidad, confianza del operador y disciplina de release.

Varios de los hallazgos de la revisión externa previa quedaron **desactualizados** por la
evolución del repo (Fases 1–6, ~40 tools, persistencia PostgreSQL, host-key TOFU): se
detallan confirmados/refutados/matizados en §4.

## 3. Tabla de hallazgos por severidad

| # | Sev | Ubicación | Hallazgo | Corrección propuesta |
|---|-----|-----------|----------|----------------------|
| R1 | **Requerido** | `ai-assistant/.../audit/AiAuditLog.kt:55-80` | **Inyección de fórmulas CSV.** `csvEscape` neutraliza `,"\n\r` pero no el prefijo de fórmula (`= + - @`). Campos como `outputTail`/`prompt`/`host` derivan de **output de equipos (no confiable)**: un banner `=HYPERLINK("http://evil",...)` queda escrito tal cual y se ejecuta al abrir el CSV en Excel/LibreOffice. | Prefijar con `'` (o `\t`) todo campo cuyo primer char ∈ `= + - @`; test con un banner malicioso. |
| R2 | **Requerido** | `ai-assistant/.../audit/AiAuditLog.kt:36-53` | **Append sin concurrencia segura.** No hay lock. Con 2 clientes MCP concurrentes (`propose_commands` simultáneos) las dos llamadas a `append()` corren la carrera del check `Files.exists`/header y el `Files.writeString(APPEND)` de filas multi-write (commands + outputTail≤800) puede intercalarse → **headers duplicados y filas corruptas**. El audit es la "verification story" del producto. | Serializar `append()` (lock de instancia o `FileChannel.lock()`), o declarar el CSV best-effort y hacer de `command_audit` (PostgreSQL, Fase 3) el sistema de registro. |
| R3 | **Requerido** | `ai-assistant/.../safety/RiskClassifier.kt:142-144` | **Fallback SAFE para verbos no catalogados.** Un comando no indentado cuyo primer token no está en ningún set cae a **SAFE**. `debug all`, `clear ip bgp *`, `test`, Junos `request system ...` (salvo `zeroize`) muestran semáforo **verde** al operador → invitan al rubber-stamp. *La invariante NO se rompe* (el gate humano igual dispara; la whitelist read-only es independiente), pero el semáforo engaña. | Cambiar el fallback no-indentado a **CONFIG**; agregar manejo de `debug`/`clear`/`test`/`request`. Test de regresión por comando. |
| R4 | **Requerido** | `mcp-server/.../protocol/McpDispatcher.kt:151,348` | **Rol auto-declarado por header.** `transport.role` sale de `X-OpenTermX-Role` (default OPERATOR). La separación multi-agente COMPLIANCE/OPERATOR es **advisoria**: un cliente puede declararse COMPLIANCE para emitir un approval token y luego OPERATOR para usarlo. *No bypassa la invariante* (el ApprovalGate humano de `propose_commands` igual dispara), pero el control de compliance no es una frontera de seguridad como está. | Documentar explícitamente la limitación, o atar el rol al token/credencial en vez del header. |
| H1 | **Requerido** (higiene) | raíz del repo | **5 scratch files commiteados:** `.claude-mcp-progress*.json` (×4) + `.claude-render-fix-progress.json`. Ruido de scratch en el árbol versionado. | `git rm` + agregar patrón a `.gitignore`. |
| H2 | **Requerido** (higiene) | raíz | **Sin LICENSE.** Sin licencia, el repo es "todos los derechos reservados" por defecto — bloquea uso/contribución. | Crear `LICENSE` (preguntar al autor qué licencia — lo cubre el PROMPT 2). |
| H3 | **Requerido** (higiene) | `.github/` ausente | **Sin CI.** Los tests no corren en push/PR: nada impide mergear un build roto. | `/.github/workflows/ci.yml`: `./gradlew check` (ya incluye pytest) en push y PR como quality gate. |
| N1 | **Nit** | `README.md` | README raíz es un hub de 8 líneas que enlaza a docs/, sin quickstart de build (`./gradlew check`, JDK 21, cómo correr). | Agregar sección "Build & test" mínima. |
| O1 | **Optional** | `mcp-server/.../handlers/InspectSessionHandler.kt:28-41` | `inspect_session` está acotado server-side a 500 líneas **y redactado**, pero 500 puede ser mucho contexto por llamada. | Considerar bajar el default (no el max) o documentar el costo de contexto. |
| O2 | **Optional** | `gradle/libs.versions.toml` | Sin escaneo de dependencias automatizado. Librerías razonablemente actuales (jsch fork mwiede 0.2.21, Jackson 2.18.1, Lucene 9.12, Javalin 6.4); sin CVE evidente. | Agregar OWASP dependency-check o el plugin `versions` al CI como mejora. |
| F1 | **FYI** | `ssh-comm/.../SshConnection.java:70-124` | **Positivo:** host-key checking implementado con TOFU + `~/.ssh/known_hosts` y default `RejectAllHostKeyVerifier` (rechaza claves desconocidas salvo verifier explícito). Refuta la preocupación de "host key checking en SSH". | — (mantener). |

**Critical: ninguno.** Se buscó activamente un camino device-mutating sin gate (todos los
handlers con `mutating=true`, los pipes de `run_readonly_command`, `run_macro`, `open_session`,
snapshots, policy, rollback, fingerprint) y la invariante se sostiene.

## 5. Checklist code-review-and-quality

- [x] **Correctitud.** Build verde, 73 pytest + unit tests pasan. Dispatcher maneja método
      desconocido (`METHOD_NOT_FOUND`), params malformados (`INVALID_PARAMS`) y excepciones
      (atrapadas → `toolCallError`/`INTERNAL_ERROR`). Sin NPE/cast inseguro evidentes en los
      caminos calientes revisados.
- [x] **Legibilidad.** KDoc denso y consistente; nombres claros; los "por qué" (decisiones,
      errores #N del catálogo) están documentados en el código. Por encima de la media.
- [x] **Arquitectura.** Separación por módulos limpia (transporte vs dominio IA vs
      persistencia vs comm). La invariante de seguridad vive en un punto (ApprovalGate) y se
      reusa. `EffectiveCapabilitiesService` pendiente (6C.1) centralizará capacidades.
- [~] **Seguridad.** Invariante central intacta. **Pendiente:** integridad del audit CSV
      (R1/R2), rol auto-declarado (R4), semáforo SAFE engañoso (R3). Redacción de credenciales
      aplicada en `inspect_session`/`propose_commands`/resources. Host-key TOFU correcto.
- [x] **Performance.** Caché TTL en enriquecimiento de perfiles; indexación Lucene incremental
      por `sourceHash`; particiones mensuales en métricas; runner con mutex por sesión. Sin
      hotspots O(n²) ni I/O en el hilo FX detectados (los diálogos usan thread propio).

## 6. Plan de corrección propuesto (commits atómicos, ~100–300 líneas)

Orden por severidad; refactors separados de fixes funcionales; cada fix de comportamiento
con su test de regresión en el mismo commit.

**Fixes funcionales (con test):**

1. `fix(audit): neutralizar inyección de fórmulas CSV` — R1. `csvEscape` antepone `'` a
   campos con primer char `= + - @`. Test: banner `=HYPERLINK(...)` en `outputTail` queda
   inerte. (~40 líneas)
2. `fix(audit): serializar append del audit CSV` — R2. Lock de instancia en `append()` +
   header atómico. Test concurrente: N hilos → N filas + 1 header, sin corrupción. (~60 líneas)
3. `fix(safety): verbos no catalogados no son SAFE` — R3. Fallback no-indentado → CONFIG;
   manejo de `debug`/`clear`/`test`/`request`. Test por comando (`debug all` → CONFIG,
   `clear ip bgp *` → CONFIG). (~80 líneas)

**Endurecimiento / documentación:**

4. `docs(security): rol MCP es auto-declarado por header` — R4. Documentar la limitación de
   la malla de roles en el README del módulo y KDoc del dispatcher; (opcional, mayor) atar el
   rol al token. (~30 líneas, sin test)

**Higiene (sin test, verificar `./gradlew check` verde):**

5. `chore: eliminar scratch files de progreso` — H1. `git rm .claude-*progress*.json` +
   patrón en `.gitignore`. (~10 líneas)
6. `chore: agregar LICENSE` — H2. **Preguntar al autor la licencia antes de crear.**
7. `ci: agregar GitHub Actions con ./gradlew check` — H3. `.github/workflows/ci.yml` en push
   y PR como quality gate. (~40 líneas)
8. `docs: quickstart de build en el README raíz` — N1. (~20 líneas)

**Mejoras opcionales (backlog):** O1 (default de `inspect_session`), O2 (dependency scanning
en CI).
